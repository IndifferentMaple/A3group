Project Overview

Topic

Group 12 aims to create a mobile application for our group project. The application will be available for iOS and Android devices. Our app will be used to report criminal incidents on the road by linking the application to a dashcam within a vehicle. The goal of our application is to create safer roads for the people of Australia to be able to drive on. The reporting system will allow everyday people to be able to send in evidence of crimes committed on the road. This will act as a deterrent for people willing to break the law, as they will have a higher chance of being caught committing these crimes if more people adopt our application. Our application will also make the life of police officers and other emergency workers easier, this is because as the number of potential criminals decreases, so does the likelihood that those emergency services will be contacted because of the criminal actions. Another goal of ours is to make the process of gathering evidence against offenders easier by giving everyday people easy access to do so through our app. By being able to deter criminals and expedite the process of detaining them we will in turn create safer roads through the use of our app.

Motivation

We used Austin Bowen’s project idea as a basis to build off as we all agreed it was a very interesting choice. Safety and trying to ensure the safety of everyday people is one of the other reasons why we chose to pursue this project. By creating this reporting app and allowing it to be used on a wide scale it will in turn become a deterrent. This will discourage criminals from breaking the law as the chance of them being caught will increase. If we were to show this to a future employer, they would gain an understanding of our group’s morals and ethics as one of our main reasons for creating this project is to make the roads safer. Our application aims to be the most innovative tool relating to road safety and crime submission, there are no other competitors that are similar to us, so we plan to capitalise off this unused concept.

Landscape

There are other crime-related apps such as Citizen [1] which allow the user to report crimes going on in the area close to them and to be able to contact first responders. The app uses a GPS to show the crimes on a real-time map in location to the user and also warns users of any dangerous crimes in their area. However this is the closest available application that is remotely similar to our project. The features listed on the Citizen app is only a minor feature within our application. Our application is innovative since there is no other application that has the same general concept on the market. This is a key difference to the app. This application will try to take advantage of an untapped demographic who will use our application Dash Cam Now as it will be more convenient in their daily use.

Aims

The aim of our project is to create a mobile application that can be used by everyday people. It will be a reporting application allowing the user to connect to their dashcam and send information straight to the police.

Progress

Where It Began

This idea was first sparked by our team member Austin, he first thought of this idea when needing to create a project idea for his Assessment 1 requirement. The first-ever drafts and features for this project were completely different to what is being developed currently. Austin’s idea was originally a scanner that would be attached or installed to motor vehicles that will detect various breaches of road rules, this scanner would automatically be recorded when drivers breach a road rule. This video would instantly be uploaded to State Transport, the defendant’s insurer, and to the authorities.

Planning

Our group’s original plan was to elaborate and expand on our team member Austin’s idea. At the time, Austin’s idea was a much more feasible project compared to all the other members who all had concepts that were much too complicated or would take too long to develop. We first started discussing how it would be possible to make this scanner functional, as the main features and overall vision were already developed by Austin beforehand. There was already a method of making this functional in Austin’s original plan, which was to have the scanner physically installed into the motor vehicle with a centralized computer inside the car as well, however, this method did not seem possible given the obvious factors that played into it. Our group researched many ways on how this would be possible, however, there was no avail. This project was at a standstill, so we all opted to go for a different route but still took the general concept and elements from Austin’s idea.

Our next plan was to use a modern dashcam, which was a much more affordable and feasible approach, and to create an API inside the dashcam which could take clips and upload them to a police database with a couple of interactions within the touchscreen. This idea seemed as a much more possible plan, as the use of APIs is not unfamiliar in this age of technology. Our group researched and discussed different methods that could help develop this plan, we contacted a platform engineer at Deloitte Digital [1] for assistance in helping this project to life by giving his input on our situation. After having many back-and-forth conversations with this IT professional, we were disappointed to learn that this idea would be an unrealistic approach for us, as having to research and use a physical item when each team member lived remotely would create challenges and conflicts. In the conversation with the IT Professional, they mentioned creating a mobile application instead, as it would be much easier to use an API to connect a dashcam and an application.

The idea of a mobile application lit the spark back into our development of the project, our group saw the many benefits that came with creating a mobile application, and it would be much more feasible given that there were many free resources that were available on the internet, also given the fact that application development is not limited to all members being present, but usually being done remotely. Our group researched and discussed how this application would work, alongside adding new features that would make this app not only limited to being a clip and submission concept. We finally decided that this approach was the best decision as it brought in many factors that each team member could contribute to.

App development is a fast-paced industry, meaning that there are many creators who will gain inspiration and inevitably end up having the same idea for an application. For this reason, we will need to obtain a patent for our application to ensure our intellectual property is protected. [1] We will need to ensure to obtain a standard patent, this includes some of our technical aspects including the: [2]

- Server-based processing systems and methods.
- User or mobile interface-based processing systems and methods.
- Interaction between the user, mobile interface, and device with the server.
- System for processing information partially at the mobile interface end and partially at the server end.
- Method and system for generating output to a mobile device.
- Creation of a database on the server or mobile device.

If obtaining a standard patent is successful, it can provide up to 20 years of protection.

With having users engage in a community-like feature such as the road report system, the hopes of building a network of information to communicate with each other will help create a more knowledgeable community that encourages to help assist others on the road.

To enable the connection of Dash Cams to our application, there will be three different methods will be used. The utilization of 4G, 5G as well WIFI [1] enables the user to stream and download videos directly to the application. The use of 4G and 5G are essential for users without access to WIFI, allowing the process to be used with 4G and 5G, which will create less hassle for the user and increase the rate of our application being used more regularly and remotely.

User Interface:

This application will have a simplistic user interface. This user interface will focus on the ease of access and concise presentation of information. Our UI will utilize a blue colour scheme. This colour scheme will link to the blue colour scheme that police in Australia use on their uniforms and logo, [1] this will give a sense of connection between our app and the proper authorities. Minimal text and graphical images will be used throughout the navigation sidebar to keep the simplicity theme throughout the application.

Navigation Panel

The user interface will consist of a navigation bar located at the bottom of the screen, each having four panels with different choices of actions. The sidebar will roughly take a maximum of 20% of space on the screen, to allow sufficient space for information to be viewed on the rest of the screen. The navigation bar will use the blue colour scheme, while the portion displaying the information will use white to not interfere with the text on the screen.

First Bar – Report/Submission

The first choice on the navigation bar will display an exclamation mark symbol, which will direct the user to the incident report page. This report page will be in a submission format where a user will be able to attach a recording from their cloud storage and submit it to the authorities. Users will have the option to submit this information anonymously or provide information when filling out the necessary details, this is to protect the user’s identity in the case that the user believes that they would be in danger if they provide their details. Furthermore, users will also have the option to provide the address of the incident on the report, linking itself to the location service mentioned, it will allow other users to view where and what incident occurred. Users will have the option of generating a QR code or link to share the video with others online, this will help encourage sharing videos between communities and users when necessary.

Second Bar – Cloud Storage

The second choice on the navigation bar will display a box symbol, which will direct the user to a page where they are able to view all their uploaded videos. The storage page allows the user to play back any footage they have collected to ensure that they do not submit any videos containing unnecessary evidence. The storage page will allow users to delete their saved footage, so they will not have large amounts of space being taken up on the cloud. A video playback feature will allow users to view their videos at a slower speed, with options ranging from 0.25x to 1x. This will give users a better focus on viewing any small details they may have missed.

Third Bar – Hotline

The third choice on the navigation bar will display a telephone symbol, which will direct the user to a page where different phone numbers are displayed. The numbers displayed are emergency service numbers, where in the case of an incident that is important or severe, easy access to various emergency services will be available. The numbers, 000 [1] , 131 444 [2] and 1800 333 000 [3] will all be featured with a small description informing the user of the purpose of the number. 000 for immediate emergencies where police, fire or ambulance services are needed, 131 444 for non-urgent police assistance and 1800 333 000 for a direct report of criminal information.

Fourth Bar – Audit Log

The fourth and final choice on the navigation bar will display a book symbol, which will direct the user to a page where a log of actions and report of previous recordings and submissions will be available. Inspired by Discord’s audit log feature, [1] the purpose is to allow the user to view their previous reports to prevent the user from potentially submitting the same report twice.

iPhone Widgets:

The application will also take advantage of the widget system offered on IOS 14. [1] When implementing a widget onto the device’s homepage, it will display the emergency services tab of our application. This will decrease the response time of having to load the app and choose the right navigation bar option.

Testing

In the early days of development, and since the team is still quite small, simplified application testing methodologies can be employed to speed up the initial time to market with a proof of concept/beta.

Once the application reaches commercial release after the initial beta testing and has reached a good number of real-world users, new features will be developed for the application. Since the development team will certainly expand for version 2 and beyond, the testing methods will need to be improved and a good project management system like Jira, could be justified for use. To ease that transition it pays to have a good plan for what the later testing will look like so that useful groundwork can be done in the early days.

Testing Level Early Days Later Advanced Development
Unit tests Wireframe/groundwork Full comprehensive development requirement
Integration Testing Big Bang Hybrid
System (Alpha) Testing Manual Automated
Beta Testing UAT / Public Forum UAT / Public Forum
Unit testing - This will be steered mainly by software programmers' recommendations. Beginning with a basic wireframe for unit testing included with the code. This will require the selection of a suitable Unit Testing framework for the software development environment that is ultimately opted for. In later development, there can be a review on the unit testing to flesh it out more and expand on the unit tests with clarity in structure and that there are rich helpful comments in the code. [1]

Integration testing for the prototype benefits from the speed of Big Bang (all together) integration testing, making it easier for the small team to achieve in a timely manner. Later development with a larger team will need to involve a more comprehensive integration testing system, like the hybrid integration testing model, which allows for both bottom-up and top-down testing of the software layers and units. [2]

System testing can be done manually at the alpha development stage of the very first version. However, it's essential to take some time to properly develop test plans and record these in such a way that AWS Device Farm can be taken advantage of for thorough cross-platform testing. By the time the project reaches the next version, AWS Device Farm will be set up and running to perform automated testing, along with a step-by-step wiki on how to add new tests when new functionality is added. [3]

Beta testing recruits for the first version will be sourced via an active facebook page and promotions for helpers made by a clever marketing person who can drive the page and create a good social media presence. Promotions will target conscientious drivers with a good balance between those who are iOS users versus Android users. Their questions, bugs and feedback can be gleaned and managed via a mediated forum like Github Issues or a Discord channel. [4]
